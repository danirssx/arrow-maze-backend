# AI Usage Log: MAZ-194 Full-catalog QA test user (backend)

## Task / Problem

QA needs one stable, documented, non-production account to test the complete level
catalog (progress, leaderboard submit, refresh-token, sync) without creating many
accounts. MAZ-194 adds that account to the seed and documents a runbook.

## Tool and Model

Claude Code / Claude Opus 4.8.

## Prompt Used

The user asked to implement `MAZ-194` following both repository `AGENTS.md` files, root
`MEMORY.md`, `Linear_MCP_Guideline.md`, a fresh worktree, AI usage logging, checks,
commit/push/PR, Linear updates, and a review of affected tickets (builds on MAZ-178's
demo-credentials seed; interacts with MAZ-191 client locking).

## Chosen Policy (Open Product Decision)

The ticket required the team to choose whether the QA account follows normal
progression or bypasses locks in local/dev. **Chosen: normal progression from level 1.**

Why (vs. the alternative):

- A local/dev **lock bypass** would touch the client sequential-locking rules (MAZ-191)
  and risks weakening normal users; the AC explicitly forbids weakening normal
  progression. Normal progression needs **no rule change** — a documented seeded account
  only, so normal users can never be affected.
- It matches the ticket title ("end-to-end level progression"): starting empty exercises
  the real first-completion + unlock path a pre-unlocked account could not.

The decision is recorded in code (`QA_PROGRESSION_POLICY = "normal-progression"`), the
README runbook, and this log; the human ratifies it at the PR/Linear review gate.

## Result Obtained

- Added the `qa_catalog` account (`qa@arrowmaze.test` / `ArrowDemo!QaCatalog`,
  documented non-secret) to `DEMO_USER_CREDENTIALS`, plus `QA_FULL_CATALOG_USER_ID` and
  `QA_PROGRESSION_POLICY`. It is auto-created and auto-covered by the existing
  validity/uniqueness/hash tests.
- Extracted `DEMO_PROGRESS` + `DEMO_LEADERBOARDS` from `seed.ts` into
  `prisma/seed-data/demoProgress.ts` (seed behaviour unchanged) so the QA account's
  empty-start policy is testable without executing the seed (which connects to a DB).
- README: QA row in the demo-credentials table + a full-catalog QA runbook (local/dev
  only, normal progression, step-by-step).

## Lessons / Limitations

- `seed.ts` runs `main()` on import, so importing it in a unit test would execute the
  seed against a DB; the progress/leaderboard data was extracted into a pure seed-data
  module to keep it testable — mirroring `demoCredentials.ts` / `authoredLevels.ts`.
- No `src/` production behaviour changed; this is seed data + docs + tests.
- The QA account exercises the full catalog by playing levels in order; a fresh
  `npm run db:setup` is required before the documented login works.
